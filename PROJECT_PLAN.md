## 1. 제품 개요

`전지적관람시점`은 공연·전시 정보를 자연어와 인덱스 검색으로 찾고, 지도 위 포스트잇 핀을 통해 주변 문화 정보를 탐색하는 모바일 우선 웹앱이다. 운영자는 Excel, CSV, 메모, 문서 등의 자료를 업로드하고, 시스템은 자료를 정해진 게시글 양식으로 자동 정리한 뒤 위치를 분석해 지도에 게시 후보로 준비한다.

핵심 경험은 다음 세 문장으로 정리한다.

1. 사용자는 첫 화면의 `pov:` 검색창에 보고 싶은 것을 자연스럽게 입력한다.
2. 검색 결과를 지도 위 포스트잇 핀과 목록으로 탐색한다.
3. 운영자는 형식이 제각각인 자료를 올리고, 자동 생성된 게시글을 검수해 발행한다.

## 5. 사용자 화면 기획

### 5.1 랜딩/검색 화면

전체 배경은 흰색이며 모든 핵심 요소는 가로 중앙 정렬한다. 장식은 최소화하고 검은 글자와 충분한 여백으로 브랜드를 강조한다.

위에서 아래 순서는 다음과 같다.

1. POV 로고
   - 원본: `/workdata/povlogo.png` (1000×1000, 투명 PNG)
   - 구현 시: `/webapp/frontend/public/logo.png`
   - 모바일 표시 크기 권장: 120~152px
   - 데스크톱 표시 크기 권장: 160~200px
2. 브랜드 타이틀
   - 문구: `전지적관람시점`
   - 검은색, 굵은 글자, 중앙 정렬
3. 검색 영역
   - 왼쪽 고정 레이블: `pov:`
   - 한 줄 검색창
   - 검색 실행 버튼 또는 Enter 입력
   - 검색창은 키워드와 자연어 질문을 모두 받는다.
4. 아래 방향 화살표 아이콘
   - 검색 결과 또는 지도 영역으로 스크롤 이동
   - 가벼운 위아래 모션으로 다음 콘텐츠가 있음을 표현
5. 지도/검색 결과 영역

검색 예시는 다음과 같다.

- `이번 주말 무료 전시`
- `성수동 주차 가능한 사진전`
- `아이와 함께 볼 수 있고 도슨트가 있는 전시`
- `작가 김OO 전시`

### 5.2 지도 탐색 화면

- 모바일에서는 화면 대부분을 지도로 사용하고 결과를 하단 시트로 표시한다.
- 데스크톱에서는 왼쪽 결과 목록, 오른쪽 지도의 2단 구성을 기본으로 한다.
- 사용자는 드래그로 지도를 이동하고, 버튼·핀치·휠로 확대/축소한다.
- 현재 보이는 지도 영역이 바뀌면 `이 지역 검색` 버튼을 제공한다.
- 핀이 많을 때는 군집 숫자로 묶고 확대 시 개별 핀으로 분리한다.
- 핀은 포스트잇을 연상시키는 사각형 형태로 제공하되 지도 가독성을 해치지 않도록 크기를 제한한다.
- 핀에는 대표 분류, 짧은 전시명 또는 상태를 표시할 수 있다.
- 마감 임박, 무료, 도슨트 있음 등의 상태는 작은 배지로 구분한다.

### 5.3 핀 선택과 게시글 보기

포스트잇 핀을 누르면 다음 방식으로 게시글을 연다.

- 모바일: 하단 미리보기 카드 → 위로 확장하거나 `자세히 보기`로 전체 상세 진입
- 데스크톱: 지도 옆 상세 패널 또는 중앙 모달
- URL은 게시글별 고유 주소를 가져 새로고침과 공유가 가능해야 한다.

미리보기에는 대표 이미지, 전시명, 기간, 장소, 관람료를 우선 노출한다. 상세에서는 고정 양식 전체와 사진 갤러리를 보여준다.

### 5.4 반응형 원칙

| 구간 | 기본 레이아웃 |
|---|---|
| 0~767px | 단일 열, 모바일 하단 시트, 터치 우선 |
| 768~1199px | 넓은 단일 열 또는 지도+축약 패널 |
| 1200px 이상 | 결과 목록과 지도의 2단 레이아웃 |

- 터치 대상은 최소 44×44px을 확보한다.
- 지도와 검색 결과 사이를 탭으로 전환할 수 있게 한다.
- 검색창, 주요 버튼, 관리자 입력 화면은 모바일 키보드가 열려도 가려지지 않아야 한다.
- 이미지에는 대체 텍스트를 제공하고 키보드 탐색과 명확한 포커스 표시를 지원한다.

## 6. 검색 설계

### 6.1 인덱스 검색

제목, 작가, 장소, 전시 내용, 감상평, 주변 정보 등에서 빠르게 일치 결과를 찾는다.

- PostgreSQL 전문 검색과 유사 문자열 검색 사용
- 날짜, 관람료, 지역, 도슨트, 주차 여부 필터
- 현재 지도 중심과의 거리 가중치
- 전시 중인 게시글, 가까운 거리, 제목 일치도를 우선 순위로 반영

### 6.2 AI 검색

자연어 질문을 구조화된 검색 조건으로 변환한다. AI가 게시물을 직접 지어내는 방식이 아니라, 저장된 게시물 중 조건에 맞는 결과를 찾고 간단한 선택 이유를 생성한다.

처리 예시:

```text
입력: 이번 주말에 강남에서 무료로 볼 수 있고 주차 가능한 전시

구조화 조건:
- date_range: 이번 주말
- area: 강남
- admission_fee: 무료
- parking: 가능
```

- 검색 결과에는 원본 게시글을 반드시 연결한다.
- 결과가 없으면 조건을 임의로 충족한 것처럼 답하지 않고 조건 완화를 제안한다.
- AI 장애 시 인덱스 검색으로 자동 전환한다.
- 검색어와 결과 클릭 데이터를 익명화해 검색 품질 개선에 사용한다.

## 7. 게시글 데이터 형식

### 7.1 운영자에게 보이는 단일 본문 양식

운영자 입력 화면은 여러 개의 분리된 텍스트 필드를 강요하지 않는다. 하나의 본문 편집기에 아래 고정 텍스트를 미리 넣고, 각 항목 뒤에 내용을 작성한다.

```markdown
전시명:

작가(작가소개):

관람료:

전시기간:

장소:

도슨트(전시장 가이드) 유무:

찾아가는 방법:

주차정보:

전시내용:

굿즈샵정보:

주변에 함께 볼 만한 전시:

주변에 볼거리:

맛집:

감상평:

페르소나 정보입력:
```

사진은 본문 중 원하는 위치에 삽입할 수 있고, 별도로 대표 이미지를 하나 지정한다.

### 7.2 내부 저장 방식

편집 경험은 단일 본문을 유지하지만 검색, 지도, 날짜 필터를 위해 서버는 본문에서 구조화 메타데이터를 자동 추출한다.

| 데이터 | 용도 |
|---|---|
| `body_markdown` | 운영자가 편집하는 원문 전체 |
| `metadata_json` | 전시명, 작가, 기간, 관람료, 도슨트, 주차 등의 추출 결과 |
| `search_vector` | 전문 검색용 인덱스 |
| `embedding` | 의미 기반 AI 검색용 벡터 |
| `address` | 원문에서 추출·확정된 장소 주소 |
| `location` | PostGIS 지도 좌표 |
| `images` | 대표 이미지와 본문 이미지 |
| `source` | 직접 입력, Excel, 메모, API 등의 출처 |
| `status` | 초안, 검수 필요, 예약, 공개, 보관 |

운영자가 본문을 저장하면 메타데이터를 다시 추출한다. 자동 추출값의 신뢰도가 낮거나 주소가 모호하면 `검수 필요` 상태로 남긴다.

## 8. 숨겨진 관리자 진입과 로그인

### 8.1 진입 동작

- 랜딩 화면의 POV 로고를 짧은 시간 안에 3회 연속 누르면 관리자 로그인 페이지로 이동한다.
- 권장 판정 시간: 첫 탭부터 1.5초 이내 3회
- 모바일 탭과 데스크톱 클릭을 동일하게 처리한다.
- 일반 사용자에게 관리자 링크를 별도로 노출하지 않는다.

### 8.2 보안 원칙

로고 3회 탭은 메뉴를 숨기는 장치일 뿐 인증 수단이 아니다.

- 관리자 계정과 강한 비밀번호 사용
- 비밀번호는 Argon2id 해시로 저장
- 로그인 실패 횟수 제한과 IP/계정 단위 지연
- 세션은 `HttpOnly`, `Secure`, `SameSite` 쿠키로 관리
- 운영 환경에서는 2단계 인증 적용 권장
- 모든 작성·수정·발행 작업을 감사 로그에 기록
- 관리자 API는 화면 주소를 알아도 인증 없이는 접근할 수 없어야 한다.

## 9. 관리자 화면

### 9.1 대시보드

- 공개 중, 임시 저장, 검수 필요, 업로드 처리 중, 오류 건수
- 최근 업로드와 최근 수정 게시물
- 주소 확인이 필요한 게시물
- 전시 종료 임박 또는 종료된 게시물

### 9.2 게시글 직접 작성

- 고정 양식이 들어 있는 단일 Markdown/Rich Text 편집기
- 사진 드래그 앤 드롭·모바일 사진 선택
- 대표 이미지 지정
- 추출된 메타데이터 미리보기
- 지도 위치 미리보기와 핀 위치 수동 보정
- 임시 저장, 미리보기, 공개, 예약 발행

### 9.3 파일 업로드

MVP 지원 형식:

- `.xlsx`, `.csv`
- `.txt`, `.md`
- `.docx`
- 텍스트 추출 가능한 `.pdf`
- 이미지 자료는 OCR 적용 후보로 두되 품질 검수가 필요하다.

여러 파일을 한 번에 올릴 수 있고, 파일 하나에 여러 공연·전시가 있으면 항목별 게시글 초안으로 분리한다.

### 9.4 자동 게시글 생성 파이프라인

```text
파일 업로드
  → 파일 형식·크기·악성 파일 검사
  → 텍스트/표/OCR 추출
  → 게시물 단위 분리
  → AI 양식 매핑
  → 날짜·가격·주소 정규화
  → 주소 지오코딩
  → 중복 후보 탐지
  → 본문·메타데이터·지도 핀 초안 생성
  → 운영자 검수
  → 발행
```

중요 원칙:

- AI 생성 결과는 자동 공개하지 않고 항상 `검수 필요` 초안으로 만든다.
- 원본 파일과 추출 결과를 함께 보관해 운영자가 비교할 수 있게 한다.
- 누락된 항목은 추측으로 채우지 않고 `확인 필요`로 표시한다.
- 전시명, 기간, 장소가 같은 자료는 중복 후보로 묶는다.
- 지오코딩 결과가 여러 개인 경우 지도에서 운영자가 선택한다.

## 10. 기술 아키텍처

### 10.1 프론트엔드

- Nuxt 3 + TypeScript
- Vue Composition API
- 상태 관리: Pinia
- 스타일: CSS variables 기반 자체 디자인 시스템
- 지도: MapLibre GL JS를 기본 인터페이스로 사용하고 국내 지도/지오코딩 공급자는 어댑터로 분리
- 이미지 최적화: Nuxt Image
- 폼 검증: 공유 가능한 스키마 검증 도구 사용
- 일반 공개 페이지는 SSR/SEO를 지원하고 지도·관리자 화면은 클라이언트 기능을 병행한다.

### 10.2 백엔드

- Go HTTP API
- 라우팅: `chi` 계열의 가벼운 라우터
- DB 접근: `pgx` + `sqlc`
- 마이그레이션: `goose`
- 백그라운드 작업: Redis 큐 기반 Go worker
- 파일 저장: S3 호환 스토리지, 로컬 개발은 MinIO
- 인증: 관리자 세션 쿠키 방식
- AI 제공자는 인터페이스로 추상화해 모델 교체가 가능하도록 구성

### 10.3 데이터베이스

- PostgreSQL
- PostGIS: 반경 검색, 지도 영역 검색, 거리 정렬
- pgvector: 의미 검색 임베딩
- 전문 검색 인덱스와 유사 문자열 인덱스

### 10.4 Docker Compose 서비스

```text
reverse-proxy  외부 진입점, TLS/라우팅
frontend       Nuxt 3
api            Go API
worker         파일 분석·AI 변환·지오코딩
postgres       PostgreSQL + PostGIS + pgvector
redis          작업 큐·캐시·속도 제한
minio          개발용 이미지·원본 파일 저장소
```

운영 환경에서는 데이터베이스와 객체 저장소를 관리형 서비스로 교체할 수 있게 환경 변수로 연결한다.

## 14. 비기능 요구사항

- 첫 화면은 모바일 4G 환경에서 빠르게 표시되어야 하며 지도 코드는 필요 시점에 지연 로드한다.
- 지도 이동 중 핀 요청은 디바운스하고, 현재 화면의 좌표 경계로 제한한다.
- 이미지 업로드 시 WebP/AVIF 파생본과 썸네일을 생성한다.
- 공개 상세 페이지는 검색 엔진용 메타데이터와 공유 이미지를 제공한다.
- 서버·클라이언트 오류 추적, API 지연, 업로드 실패율을 관측한다.
- 데이터베이스와 객체 저장소는 정기 백업한다.
- 개인정보와 관리자 자격 증명을 로그에 남기지 않는다.

## 15. 개발 단계

### 1단계 — 기반 구축

- Nuxt 3, Go API, PostgreSQL, Redis, MinIO 구성
- Docker Compose 한 번으로 전체 실행
- DB 마이그레이션과 기본 CI

### 2단계 — 공개 화면

- 흰 배경 랜딩, 로고, 타이틀, `pov:` 검색창
- 반응형 지도와 포스트잇 핀
- 게시글 상세와 인덱스 검색

### 3단계 — 관리자

- 로고 3회 탭 진입
- 인증, 단일 본문 편집기, 이미지 업로드
- 지도 좌표 미리보기와 게시 상태 관리

### 4단계 — 자동 수집과 AI

- Excel/CSV/TXT/Markdown/DOCX/PDF 업로드
- 본문 양식 자동 생성, 메타데이터 추출, 지오코딩
- 검수함, 중복 탐지, AI 검색

### 5단계 — QA와 운영 준비

- 모바일·태블릿·데스크톱 브라우저 검증
- 접근성, 보안, 성능 테스트
- 백업·모니터링·배포 문서화

## 16. MVP 완료 기준

- `docker compose up`으로 프론트, API, DB, worker가 실행된다.
- 모바일과 데스크톱에서 흰 배경의 중앙 정렬 랜딩이 동일한 브랜드 구조로 보인다.
- 키워드 검색과 자연어 검색이 실제 저장된 게시물만 반환한다.
- 지도를 이동·확대·축소할 수 있고 현재 영역의 게시글 핀이 표시된다.
- 포스트잇 핀을 눌러 게시글 상세를 볼 수 있다.
- 로고를 3회 누르면 관리자 로그인으로 진입하되, 인증 없이는 관리자 API에 접근할 수 없다.
- 운영자가 단일 양식 본문과 사진으로 게시물을 작성하고 발행할 수 있다.
- Excel 또는 메모 파일을 올리면 고정 양식의 초안과 지도 위치 후보가 생성된다.
- AI 결과의 누락·주소 모호성·중복 가능성을 운영자가 확인한 뒤 공개할 수 있다.
